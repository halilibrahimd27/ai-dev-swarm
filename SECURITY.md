# Security policy

## Supported versions

ai-dev-swarm follows simple "main branch only" support. Security
fixes land on `main`; there is no LTS or back-port channel. Pin to
a specific commit for production use.

| Branch | Supported |
| --- | --- |
| `main` (HEAD) | ✅ |
| anything else | ❌ |

## Reporting a vulnerability

If you find a security issue that should NOT be filed as a public
GitHub issue:

1. Email the maintainer at the address in the project's
   `pyproject.toml` (or in the latest `LICENSE` header).
2. Use the subject line `[SECURITY] ai-dev-swarm: <short summary>`.
3. Include: a description of the impact, steps to reproduce, the
   commit hash you tested against, and any proof-of-concept.

The maintainer aims to acknowledge within 5 business days. Public
disclosure happens after a fix is on `main`, with credit to the
reporter (unless you ask otherwise).

## Untrusted generated code

This is the load-bearing security note for ai-dev-swarm:

> The Developer and Tester roles use the Claude Agent SDK to write
> and execute code. That code is **untrusted by construction** —
> it is generated by an LLM. ai-dev-swarm runs it in a per-project
> persistent git workspace and gates every commit through a CI
> pass. Operators should **never run the orchestrator on a host
> that holds production secrets** beyond what's listed in `.env`.

If a project's generated code calls a shell, the SDK's `allowed_tools`
list bounds what shell commands it can use. The Developer is allowed
`Bash`; the Tester is restricted to `Bash(pytest:*)`. Both have
`WebFetch` and `WebSearch` explicitly disallowed (see
`src/aidevswarm/tools/claude_agent_sdk_tool.py::_DISALLOWED_TOOLS`).

## Outbound redaction (Phase 5 guarantee)

Every message that leaves the orchestrator via SSE or Telegram
passes through `SecretRedactor` (`src/aidevswarm/observability/redactor.py`).
The default patterns catch:

- Anthropic API keys (`sk-ant-...`)
- OpenAI-style keys (`sk-...`)
- GitHub PATs (classic `ghp_...` + fine-grained `github_pat_...`)
- Slack bot tokens (`xoxb-...`)
- JWTs (three-segment `eyJ...`)
- Telegram bot tokens (`<digits>:<secret>`)

Each match is replaced by `[REDACTED:<kind>]`. Add your own
patterns via `AIDEVSWARM_REDACT_PATTERNS` (comma-separated regex
list).

The redaction is the **last line of defense** — the orchestrator
also avoids logging secrets in the first place (Pydantic
`SecretStr`, no `print`-of-Settings, etc.).

## Network surface

- **FastAPI**: bound to `127.0.0.1` on the host. `0.0.0.0` inside
  the container is a *docker networking* requirement; the
  host-side guarantee comes from the docker publish line
  (`127.0.0.1:8080:8080` in `docker/compose.yml`).
- **Telegram**: outbound polling only. No webhook, no public URL.
- **Postgres / Redis**: published on host ports (`POSTGRES_HOST_PORT`,
  `REDIS_HOST_PORT`). Defaults to host loopback when `docker compose`
  is used. Operators wanting LAN access must opt in explicitly.
- **Phoenix**: published on `:6006`. No authentication — operators
  who want strict loopback should edit `docker/compose.yml` to use
  `127.0.0.1:6006:6006`.

## Kill switches

Two layers (see [THREAT_MODEL.md](THREAT_MODEL.md)):

1. **Per-project**: `aidevswarm:kill:<project_id>` in Redis. Trip
   via the web UI, the Telegram bot, or any process with Redis
   access on the operator's host.
2. **Global**: `aidevswarm:kill_switch` in Redis. Same surface,
   stops the whole orchestrator.

Both are checked every orchestrator tick (`Tick.advance_project`).

## Credentials in CI

The repo's GitHub Actions workflow runs `make verify` only. It
does NOT have access to any LLM provider — there are no
`ANTHROPIC_API_KEY` / `GITHUB_TOKEN` secrets in the workflow.
Integration tests that need an API key (`@pytest.mark.anthropic`)
skip automatically when the env var is empty.
